# OOP Achievements and Java Logic Implementation

This document details how specific Object-Oriented Programming (OOP) principles and advanced Java logic were achieved in the Modern Banking System.

## 1. Achieving OOP Principles

### Encapsulation (Data Hiding & Protection)
- **How Achieved**: All model classes (`User`, `Account`, `Loan`, `Transaction`) have private fields. Access is only possible via public getter and setter methods.
- **Why it matters**: It prevents external classes from directly modifying critical data (like account balance) without going through validation logic.
- **Logic Example**: In `Account.java`, the `balance` is private. Only the `BankingController` can update it after verifying the transaction.

### Inheritance (Reusability & Hierarchy)
- **How Achieved**: We created a `BaseFrame` abstract class that extends `JFrame`.
- **Logic Example**: `LoginScreen`, `AdminDashboard`, `CustomerDashboard`, and `EmployeeDashboard` all extend `BaseFrame`.
- **Achievement**: Common UI styling, sidebar creation, and window settings are defined once in `BaseFrame` and inherited by all screens.

### Abstraction (Simplification of Complexity)
- **How Achieved**: Using abstract methods and interfaces.
- **Logic Example**: `BaseFrame` defines the structure of a dashboard. The `DataStore` abstracts the database complexity; the UI doesn't need to know if data comes from a file or SQL.
- **Achievement**: The developer interacts with a simple `DataStore.getUsers()` method rather than complex SQL queries.

### Polymorphism (Flexibility)
- **How Achieved**: Overriding methods and using common types.
- **Logic Example**: Different dashboards override the `initUI` logic of the system while sharing the same `BaseFrame` type.

## 2. Java Logic and Design Patterns

### Model-View-Controller (MVC)
- **Model**: POJOs (Plain Old Java Objects) in `com.banking.model` represent the data.
- **View**: Swing components in `com.banking.view` handle the presentation.
- **Controller**: Logic in `com.banking.controller` processes requests and updates models.
- **Benefit**: This separation allows us to change the UI (View) without touching the database logic (Model).

### Singleton Pattern (Conceptually)
- **Implementation**: The `DataStore` and `DatabaseConnection` use static methods to ensure a centralized point of access to data and database resources.

### JDBC Logic (Persistence)
- **Implementation**: Used `PreparedStatement` to prevent SQL Injection attacks.
- **Logic**: Implemented "Upsert" (Insert on Duplicate Key Update) logic in `DataStore` to synchronize objects with the database seamlessly.

### Security Logic
- **Strike System**: `ATMController` tracks `loginAttempts`. After 3 failed attempts, it calls `updateAccountStatus(id, "BLOCKED")`.
- **Validation**: Passwords and PINs are compared using strict equality before granting session access.

## 3. GUI Development
- **Visuals**: Used `GridBagLayout` for complex positioning and `CardLayout` for seamless screen switching without window flickering.
- **Custom Styling**: Achieved a modern dark-theme look using custom colors (`Color(44, 62, 80)`) and Segoe UI fonts.
